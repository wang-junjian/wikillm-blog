# macOS 与 Apple Silicon

本聚合页沉淀作者在 macOS 平台上长达八年的技术实践（2018–2026），覆盖系统操作、包管理、Apple Silicon 转型适配、机器学习框架部署、本地大模型推理、计算机视觉项目实战、Automator 快捷操作批量自动化，以及图像/音视频处理的批量工具链。

## 一、macOS 基础与系统维护

macOS 作为整个实践体系的基座，其命令行能力与系统机制贯穿全部文章。[[macOS-系统操作|macOS 系统操作]] 汇集了窗口管理、截图、文件操作等高频快捷键，以及 `find`、`sips`、`du`、`chown` 等终端命令——这些工具是后续所有实践的前置技能。[[Finder-与终端联动|Finder 与终端联动]] 一文中沉淀的"在当前文件夹打开终端"与"清理打开方式残留"技巧，构成日常维护工作流。[[终端命令速查|终端命令速查]] 与 [[macOS-快捷键|macOS 快捷键]] 共同沉淀了这套操作体系。

macOS 的后台服务由 [[launchd-与后台服务管理|launchd]] 驱动。理解 Daemon/Agent 区别以及五类 plist 存储路径（`~/Library/LaunchAgents`、`/Library/LaunchDaemons` 等）是系统管理的关键。[[macOS-登录项清理|登录项清理]] 实践通过 `sudo find` 定位并删除残留 plist 文件，这套 find 命令模式与 [[macOS-系统操作|macOS 系统操作]] 中清理 `.DS_Store` 的思路一脉相承。[[macOS-权限管理|macOS 权限管理]] 与 [[plist-文件配置|plist 文件配置]] 是这套机制的理论基础。

远程访问方案在 macOS 生态中有多条路径。[[SSH-X11-Forwarding|SSH X11 Forwarding]] 借助 XQuartz 作为 X Server，通过 `ssh -Y` 可信转发或配置 `~/.ssh/config` 指定 xauth 路径，实现远程 Linux 图形程序本地化运行；与 [[macOS-系统操作|macOS 系统操作]] 中收录的 Microsoft Remote Desktop 形成"轻量 CLI 转发 vs 完整桌面协议"的权衡（Trade-off）。

容器化开发环境依赖 [[Docker-Desktop-排障|Docker Desktop 排障]] 经验。[[macOS-Docker|macOS Docker]] 一文中，`com.docker.diagnose check` 诊断工具定位到 Docker.raw 达 60GB 而 GC 阈值 100GB 导致无法回收的根因，通过完整清理后重装并配置 GC（20GB）、registry-mirrors、Rosetta 仿真完成恢复——这套诊断思路与 [[launchd-与后台服务管理|launchd]] 后台进程管理经验形成"应用—底层"呼应。[[Docker-镜像源配置|Docker 镜像源配置]] 与 [[Rosetta-2-转译层|Rosetta 2 转译层]] 是跨平台兼容的关键组件。

开发工具链的最新成员是 [[Claude-Code-安装指南|Claude Code 安装指南]]。该文梳理了 Native（`curl | bash`）、Homebrew cask、NPM（已废弃）三种安装路径及对应卸载方案，版本目录位于 `~/.local/share/claude/versions`，其 Native 安装的手动补救思路与 [[Homebrew-包管理|Homebrew 排错]] 思路高度相似。[[CLI-工具安装模式|CLI 工具安装模式]] 一文总结了这三种路径的共性与差异。

## 二、Homebrew 包管理

[[Homebrew-包管理|Homebrew]] 在 macOS 上负责构建整个软件生态。[[命令-brew|命令 brew]] 一文沉淀了自动更新卡死的两种解法：关闭自动更新（`export HOMEBREW_NO_AUTO_UPDATE=1`）或替换清华镜像源；同时收录了 version.rb 因 NilClass 崩溃的手动修复、Portable Ruby 校验失败的处理。brew cask 子命令延伸了 Homebrew 的能力边界——[[SSH-X11-Forwarding|SSH X11 Forwarding]] 中 `brew cask install xquartz`、[[Claude-Code-安装指南|Claude Code 安装指南]] 中 `brew install --cask claude-code`、[[macOS-Docker|macOS Docker]] 中 Docker Desktop 安装，都依赖这一扩展机制。

Homebrew 与 macOS 生态的深度融合体现在多个层面：[[M2-Max-开发环境搭建|M2 Max 开发环境搭建]] 中 brew 安装 cmake/automake/libtool 编译工具链；[[Apple-Silicon-软件安装|Apple Silicon 软件安装]] 中 brew 安装 ImageMagick、jq、rar；[[FFmpeg-音视频处理|FFmpeg 音视频处理]] 与 [[Whisper-语音识别|Whisper 语音识别]] 中 brew 安装的 ffmpeg 承担音视频转换；[[Automator-快捷操作系列|Automator 快捷操作系列]] 中 brew 安装的 unrar、webp、imagemagick 驱动批量自动化流程。[[Homebrew-镜像源配置|Homebrew 镜像源配置]] 与 [[Homebrew-cask-应用管理|Homebrew cask 应用管理]] 沉淀了包管理的最佳实践。

## 三、Apple Silicon 开发环境

Apple Silicon 转型是整个实践体系的分水岭。[[M2-Max-开发环境搭建|M2 Max 开发环境搭建]] 作为基础设施篇，在 MacBook Pro M2 Max（64GB/2TB）上从零搭建完整开发环境：通过 `scutil --set HostName` 修改主机名，安装 HomeBrew 并配置清华/阿里云镜像源，安装 Miniconda 并尝试 `.condarc` 镜像源，配置 pip 镜像源，安装 Docker Desktop 并配置 `~/.docker/daemon.json` 镜像源，创建 labelImg 与 Label Studio 标注工具环境。

[[Apple-Silicon-软件安装|Apple Silicon 软件安装]] 完善整机软件生态。关键一步是安装 Rosetta 2（`softwareupdate --install-rosetta`）以兼容 x86 软件；通过 HomeBrew 安装 ImageMagick、jq、rar 等开发工具；安装 asitop 监控 Apple Silicon 性能；安装 Chrome 插件、Oh My Zsh、V2rayU、FileZilla、Parallels Desktop 等办公与开发软件。

Apple Silicon 的硬件加速能力是机器学习框架争夺的焦点。[[TensorFlow-Apple-Silicon|TensorFlow Apple Silicon]] 通过 `conda install -c apple tensorflow-deps` 安装 Apple 优化依赖，再 `pip install tensorflow-macos` 与 `tensorflow-metal` 启用 GPU，使用 CIFAR-100 + ResNet50 训练对比 CPU/GPU 性能。[[PyTorch-Apple-Silicon|PyTorch Apple Silicon]] 则利用 MPS（Metal Performance Shaders）后端——`torch.backends.mps.is_available()` 检测可用性，`torch.device("mps")` 启用加速，同样基于 CIFAR-10 数据集和自定义 CNN 网络训练。TensorFlow 的 Metal 插件与 PyTorch 的 MPS 后端构成 Apple Silicon 上两大主流框架的 GPU 加速方案权衡（Trade-off）。

[[OpenVINO-Apple-Silicon|OpenVINO Apple Silicon]] 从源码手动编译 OpenVINO 运行时，探索对 Intel MYRIAD（Neural Compute Stick 2）的支持：brew 安装 cmake/automake/libtool/libusb 编译工具链，`git clone --recurse-submodules` 克隆仓库，cmake 配置与并行编译（`cmake --build . --parallel $(sysctl -n hw.ncpu)`），安装至 `/opt/openvino`，通过 `hello_query_device` 查看可用设备（CPU/MYRIAD），尝试将 2022.1.0 版本的 MYRIAD 插件移植到 2023 版本（未成功）。

## 四、本地 LLM 与语音推理

本地大模型推理在 Apple Silicon 上沉淀了完整实践。[[LLaMA-本地推理|LLaMA 本地推理]] 涵盖从模型下载、格式转换（GGML/GGUF）、量化到推理和 API 服务的完整链路：LLaMA 7B/13B/30B/65B 模型对比；GGUF 格式作为 GGML/GGMF/GGJT 的后继；4-bit 量化（q4_0）压缩体积；困惑度（PPL）评估量化质量；k-quants（Q2_K–Q6_K）多档位权衡（Trade-off）；llama.cpp 在 Apple Silicon 的 NEON/MPS 加速；`CMAKE_ARGS="-DLLAMA_METAL=on" pip install llama-cpp-python` 启用 Metal 后端；`python -m llama_cpp.server` 暴露 OpenAI 兼容的 REST API。

[[Whisper-语音识别|Whisper 语音识别]] 在 M2 Max 上对比测试 OpenAI Whisper 与 whisper.cpp 两款方案的速度与内存占用：QuickTime 录制 m4a，[[FFmpeg-音视频处理|FFmpeg]] 转换为 16kHz 单声道 wav；conda 创建 whisper 环境，从 GitHub 源码安装 OpenAI Whisper，测试 tiny/base/small/medium/large 五种模型；whisper.cpp 通过 `git clone` + `make` 编译，下载 ggml 模型，使用 `./main -f` 推理——对比表显示 whisper.cpp 速度约为 OpenAI Whisper 的 6 倍、内存约为其 50–60%。[[语音识别方案选型|语音识别方案选型]] 与 [[GGUF-模型格式|GGUF 模型格式]] 是这套实践的理论基础。

## 五、计算机视觉实战

[[YOLOv8-目标检测|YOLOv8 目标检测]] 构成计算机视觉实践的主线。[[Ultralytics-YOLOv8|Ultralytics YOLOv8]] 覆盖 Docker 环境搭建、训练、评估、预测、导出及多格式基准测试：`yolo` CLI（TASK MODE ARGS）统一入口；COCO128 数据集验证；Comet/ClearML 训练可视化；多格式导出（PyTorch、ONNX、OpenVINO、TensorRT、CoreML、TF Lite、PaddlePaddle 等）的 GPU vs CPU benchmark。

数据集准备依赖云端平台。[[Roboflow-快速入门|Roboflow 快速入门]] 提供工作区/项目协作、Train/Test Split、Preprocessing、Augmentation 配置、多格式数据集导出、Roboflow Universe 社区与 Python SDK 推理；[[Ultralytics-Hub-快速入门|Ultralytics Hub 快速入门]] 则提供数据集目录结构（train/valid/test）、`data.yaml`（nc、names）、Colab 远程训练与 API Key 集成——Roboflow 与 Ultralytics Hub 形成云端数据管理的两种路径权衡（Trade-off）。

自建推理服务是云端部署的替代方案。[[Ultralytics-Serving-FastAPI|Ultralytics Serving FastAPI]] 基于 FastAPI + ONNXRuntime 构建：FastAPI 项目结构（app/routers/utils/tests）；ONNX 推理（cv2 resize + transpose + InferenceServer）；UploadFile 读取与 cv2.imdecode；StreamingResponse 返回图像；gunicorn 多进程；docker buildx 多架构（amd64/arm64）镜像构建；Swagger UI 离线静态资源方案——与云端方案的权衡（Trade-off）体现在数据隐私 vs 运维成本。

[[Stable-Diffusion-文本生成图像|Stable Diffusion 文本生成图像]] 引入生成式 AI 维度。基于 LAION-5B 数据集（58.5 亿 CLIP 过滤的图文对）训练，利用潜在空间（Latent Space）扩散生成图像，CLIP 嵌入实现文本-图像多模态检索；HuggingFace Diffusers 库、Diffusion Bee（macOS GUI）、AutoFaiss（向量索引）、Apple Silicon MPS 加速构成生态工具链。[[生成式-AI-工具链|生成式 AI 工具链]] 与 [[CLIP-多模态嵌入|CLIP 多模态嵌入]] 沉淀了这套生态的技术原理。

## 六、OpenVINO 推理引擎

[[OpenVINO-入门|OpenVINO 入门]] 作为系列入口，梳理 Development vs Runtime 安装、Model Optimizer（mo）、Benchmark Tool（benchmark_app）、Accuracy Checker、POT、OMZ 工具集（omz_downloader/converter/quantizer/info_dumper/data_downloader）等组件，通过 IECore 设备查询（CPU/GPU）与 Netron 模型可视化跑通 alexnet 分类与 mobilenet-ssd 检测样例。

[[OpenVINO-工作原理|OpenVINO 工作原理]] 构建理论框架：5 步工作流（计划设置→训练→转换优化→调整性能→部署）；支持框架（TF/Caffe/MXNet/Kaldi/ONNX/PaddlePaddle）；IR 格式（.xml + .bin）；Model Optimizer 转换管线与自定义层；推理引擎设备插件（CPU/GPU/VPU/GNA/Arm）；Multi-Device/Heterogeneous/Auto-Batching/Auto 设备选择；性能衡量（Throughput/Value/Efficiency/Latency）；NNCF 量化感知训练。

[[Open-Model-Zoo|Open Model Zoo]] 自动化工具与 MTCNN 人脸检测 Demo 实战：omz_downloader/converter/quantizer/info_dumper/data_downloader 工具链，`--print_all` 列出可用模型（含 yolo-v3/v4、yolox、mtcnn 等），MTCNN 三阶段级联（Proposal/Refine/Output）人脸检测。

[[OpenVINO-图像分类|OpenVINO 图像分类]] 使用 OpenVINO Runtime API（Python）进行同步/异步图像分类推理，基于 googlenet-v1 (Caffe) 模型：OpenVINO Runtime Core v2.0 API；PrePostProcessor 嵌入预处理（set_element_type/set_layout/resize/build）；同步 infer_new_request；异步 AsyncInferQueue + 回调；Top-10 类别排序；NCHW/NHWC 布局转换。

[[OpenVINO-目标检测|OpenVINO 目标检测]] 基于 SSD300 (Caffe/VGGNet) 的同步目标检测：Caffe 模型下载解压，IR 转换（FP16/FP32），PrePostProcessor 预处理，检测输出解析（[1,1,N,7]→[N,7]，含 class_id/confidence/bbox 坐标），cv2.rectangle 绘制边界框——SSD 模型与 [[Ultralytics-YOLOv8|YOLOv8]] 形成 one-stage 检测的两种实现路径权衡（Trade-off）。

[[OpenVINO-性能分析|OpenVINO 性能分析]] 作为系列收尾，聚焦推理性能分析（层耗时）与内存分析（memory-profiler）：PERF_COUNT 配置收集层指标；InferRequest.get_profiling_info()（node_name/node_type/exec_type/real_time/cpu_time/status）；自定义表格打印 profiling 数据；memory-profiler @profile 装饰器监控每行内存增量；FP16 vs FP32 内存对比（FP16 在 CPU 上内存反比 FP32 大）。[[推理性能优化|推理性能优化]] 与 [[模型量化技术|模型量化技术]] 是这套实践的核心方法论。

## 七、Automator 快捷操作

[[Automator-快捷操作系列|Automator 快捷操作系列]] 是 macOS 自动化的核心沉淀。系列采用统一范式：Shell 脚本兼容 Apple Silicon/Intel 双架构 Homebrew 环境，通过 `osascript` 弹窗错误处理与结果通知，Finder 右键"服务"菜单触发。

[[Automator-PNG-转-JPG|Automator PNG 转 JPG]] 批量将文件夹内 PNG 重命名为两位数序号并转换为 JPG：Perl 语言一行脚本顺序重命名，ImageMagick `mogrify` 格式转换，`rm` 删除原文件。[[Automator-解压-RAR|Automator 解压 RAR]] 用 `rar x` 保留路径结构解压，`pushd/popd` 安全切换目录，解决"Operation not permitted"需为 `/opt/homebrew/bin/rar` 授予"完全磁盘访问权限"，提供带日志 `/tmp/rar_automator_debug.log` 的调试版脚本。

[[Automator-PDF2JPG|Automator PDF2JPG]] 将 PDF 每页转为单独 JPG：`magick -density 300 -quality 90 -colorspace RGB` 高分辨率转换，自动在 PDF 同名目录下创建输出文件夹，页码用 `%d` 占位符，兼容 `.pdf`/`.PDF` 大小写扩展名。[[Automator-图像拼接|Automator 图像拼接]] 实现多张图片水平/垂直拼接：`osascript` 弹窗让用户选择垂直/水平，`magick ... -append`（垂直）/`+append`（水平）拼接，自动检测首图扩展名、生成时间戳命名。

[[Automator-图像转-WebP|Automator 图像转 WebP]] 批量转 WebP：遍历 `"$@"` 自参数，跳过目录与已有 WebP，`tr` 转小写扩展名判断，`cwebp -q 80` 转换并通过 `$?` 判断成功与否——直接依赖 [[WebP-格式工具|WebP 格式工具]] 一文中介绍的 cwebp 命令。[[WebP-格式工具|WebP 格式工具]] 作为理论基础篇，介绍 `cwebp -q 80` 有损压缩（推荐长截图）、`cwebp -lossless` 无损压缩、`dwebp` 反向转 PNG 的能力。

系列中唯一使用 AppleScript 的是 [[Automator-反向选择|Automator 反向选择]]：通过 `tell application "Finder"` 获取当前窗口所有对象与已选对象，遍历取差集后 `set selection to` 重新选中，需处理无边界情况——展示 Automator 除 Shell 脚本外的另一编程入口。

## 八、图像处理与音视频

[[FFmpeg-音视频处理|FFmpeg 音视频处理]] 在 Apple Silicon 上重点测试 VideoToolbox 硬件加速性能：从 evermeet.cx 下载静态编译版 FFmpeg，通过 `-hwaccel videotoolbox` 启用硬件加速，对比 h264_videotoolbox 硬编码与 libx264 软编码的资源占用（硬编码 CPU 仅 33%、耗时 18s，远优于软编码的 1080%、40s），总结最佳实践（硬编码视频 + 软编码音频），梳理 Intel QSV / Nvidia NVENC / AMD AMF / Apple VideoToolbox 各平台硬件加速方案。

[[Mac-屏幕素材处理|Mac 屏幕素材处理]] 从截图合成 GIF 到 FFmpeg 视频智能去重压缩的完整工作流：`magick -delay 200 -loop 0` 合成 GIF；`mpdecimate+setpts` 抽帧去重；三种策略（保留音频/剥离音频/可变帧率 VFR）权衡同步与压缩；`libx264/-crf 23/-vsync cfr/-pix_fmt yuv420p` 恒定帧率万能模板；`setpts=1/N*PTS` + `atempo=N` 实现 N 倍速快进（注意 atempo 0.5–2.0 限制需串联）。

ImageMagick 是贯穿多篇的核心工具：[[Automator-PNG-转-JPG|Automator PNG 转 JPG]] 的 `mogrify`、[[Automator-PDF2JPG|Automator PDF2JPG]] 的 `magick`、[[Automator-图像拼接|Automator 图像拼接]] 的 `append`、[[Mac-屏幕素材处理|Mac 屏幕素材处理]] 的 GIF 合成——与 [[FFmpeg-音视频处理|FFmpeg]] 共同构成"图像处理工具链"互补。[[ImageMagick-图像处理|ImageMagick 图像处理]] 一文沉淀了这套工具链的完整用法。

## 十、浏览器内 AI 开发

[[中国象棋-AI-开发|中国象棋 AI 开发]] 是博客中唯一的纯前端游戏 AI 实践，用单文件 HTML 实现人机对弈：10×9 二维数组棋盘 + `{t,s}` 棋子表示；七种棋子走法规则（蹩马腿、塞象眼、隔山打牛、飞将照面）；Make/Unmake 模式避免深拷贝；Minimax + Alpha-Beta 剪枝 + 吃子优先走法排序 + 终局深度惩罚；棋子价值表 + 位置评估表双维度评估；Canvas 响应式缩放需做坐标物理像素换算；CSS 布局踩坑（flex stretch、Canvas 撑爆窄屏、媒体查询断点、全局样式覆盖遗漏）。[[Minimax-与-Alpha-Beta-剪枝|Minimax 与 Alpha-Beta 剪枝]] 与 [[Canvas-前端渲染|Canvas 前端渲染]] 是这套实践的核心技术。该实践与 [[LLaMA-本地推理|LLaMA 本地推理]] 形成"浏览器内 AI vs 本地大模型"的两种部署路径权衡（Trade-off），同时共享"零依赖、单文件、双击即玩"的极简发布理念。
