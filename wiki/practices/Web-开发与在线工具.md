## Web 基础工具链

### 静态站点与博客生成

[Jekyll](https://jekyllrb.com/) 是 Ruby 生态中最经典的静态站点生成器，以 Markdown 为核心内容格式，驱动了 GitHub Pages 早期的博客生态。作者在 [[Welcome-to-Jekyll|2020 年初探 Jekyll]] 时记录了 macOS 升级 Sonoma 后 bundler 兼容性问题，最终采用 Docker 容器化方案运行 `jekyll serve`——这是静态站点工具链在本地环境适配上的典型权衡（Trade-off）。参见 [[静态站点与内容管理]]。

构建个人博客的另一种思路是研究行业标杆的内容组织方式。[[Simon-Willison-博客|Simon Willison 的博客]] 使用 Django 自建，融合 Entries、Links、Quotes、Notes、TILs、Tools 六种内容类型，通过 SQL `UNION ALL` 合并为统一时间流，以标签为唯一组织维度。其设计哲学是"极简排版 + 高频碎片更新"，与 Jekyll 的长文博客形态形成对照。详见 [[Simon-Willison-博客分析|博客设计研究]] 与 [[GitHub-Pages-Giscus|Giscus 评论方案]]。

### 编程语言入门

开发者的技术栈扩展往往从工具链安装开始。[[Rust-入门|Rust]] 通过 `rustup` 管理版本、`cargo` 负责构建与依赖，工具链设计类似 npm/pip/go 的集合体。[[TypeScript-入门|TypeScript]] 则通过 `tsc` 编译器与 `ts-node` 运行时两层机制，为 JavaScript 引入静态类型系统——其 `tsconfig.json` 的 `target` 与 `strict` 选项是项目质量的基础守门。

[[Next.js|Next.js]] 作为 React 全栈框架，`create-next-app` 脚手架内置 ESLint 配置，`next lint` 命令提供开箱即用的代码规范检查。这些工具链的共同特征是：CLI 驱动、零配置优先、IDE 友好。参见 [[Next-AI-Draw.io|Next.js 在 AI 图表工具中的应用]] 与 [[Vercel-AI-SDK|Vercel AI SDK]]。

### 爬虫与数据采集

[[Python-爬虫实践|Python 爬虫]] 实践中一个经典坑点：豆瓣等站点对无 `User-Agent` 的请求返回 418 状态码。解决方案是在 `requests.get()` 中加入浏览器级请求头。这是 Web 开发中"浏览器模拟"基础知识的实战映射。相关技术也可参考 [[浏览器-TTS-开发经验|浏览器原生 API 实战]] 中关于浏览器能力差异的讨论，以及 [[Web-标准与协议]]。

## AI 编码智能体架构

### Cline：VS Code 原生的 Agent Loop

[[Cline-架构设计|Cline]] 是最早将 Agent Loop 嵌入 VS Code 的编码助手之一。其架构分为五层：表示层（React Webview + Ink CLI）、应用层（Extension 入口）、领域层（Controller + Task + Prompts）、基础设施层（Services + Host Bridge）、数据存储层（SQLite + 文件系统）。

核心通信采用 gRPC/Protobuf 风格，消息类型定义在 `src/shared/proto/`，通过 VS Code 消息传递通道序列化。Task 执行引擎遵循"User Input → Controller → API Call → Tool Executor → User Approval"的循环模式，StateManager 负责全局状态持久化。Cline 的 MCP（Model Context Protocol）集成是其可扩展性的关键——通过 McpHub 动态注册第三方工具。参见 [[MCP-协议生态]]、[[Claude-Code|Claude Code 的插件架构]] 与 [[AI-智能体架构]]。

### Vercel AI SDK：提供商无关的 AI 抽象层

[[Vercel-AI-SDK|Vercel AI SDK]] 解决的核心问题是：让同一份代码在 OpenAI、Anthropic、Google、AWS Bedrock 等 50+ 提供商之间零成本切换。其架构核心是 `LanguageModelV4` 接口——所有提供商必须实现 `doGenerate()` 和 `doStream()` 两个方法。

SDK 提供 `generateText` / `streamText` 两个核心函数，内置工具调用循环（Tool Loop）。更上层的 `ToolLoopAgent` 类封装了"LLM 调用 → 工具执行 → 再次调用"的完整 Agent 模式。UI 层通过 `useChat` Hook 实现框架无关的流式聊天体验，支持 React、Svelte、Vue、Angular。参见 [[Next-AI-Draw.io]] 与 [[Kilo-Code]]。

Vercel AI SDK 的 Monorepo 结构（核心 `packages/ai/` + 50+ 提供商包）与 Changesets 版本管理，是大型 TypeScript 库模块化设计的行业标杆。该 SDK 也是 [[Kilo-Code|Kilo Code]] 和 [[Next-AI-Draw.io|Next AI Draw.io]] 的底层 AI 抽象层。

### Next AI Draw.io：AI + 图表的工程范本

[[Next-AI-Draw.io|Next AI Draw.io]] 展示了一个完整的 AI 驱动 Web 应用架构：Next.js 16 + React 19 前端、Vercel AI SDK 作为 AI 抽象层、DynamoDB 配额管理、Langfuse 可观测性追踪。其核心设计是 4 个工具（display_diagram / edit_diagram / append_diagram / get_shape_library）的精细化分工——将"创建"与"编辑"分离，用 append 处理输出截断，用 shape_library 支持图标探索。

项目支持 15+ AI 提供商，通过 `lib/ai-providers.ts` 中的策略模式统一创建入口。响应缓存（`lib/cached-responses.ts`）对常见图表模板（AWS/Azure/GCP 架构图）提供毫秒级响应，是 API 成本优化的典型手段。详见 [[AI-开源项目评估]] 中对 [[Next-AI-Draw.io|Next AI Drawio]] 的评估。

### Claude Code：Anthropic 的终端智能体

[[Claude-Code|Claude Code]] 是 Anthropic 官方的终端编码智能体，核心架构包括：用户交互层（自然语言命令解析）、上下文管理系统（代码库理解 + 对话历史）、插件运行时环境、工具调用框架（Bash + Git + GitHub API + MCP）、Claude API 层。

其插件系统遵循标准结构：`plugin.json` 元数据 + `commands/` + `agents/` + `skills/` + `hooks/` + `.mcp.json`。事件驱动架构涵盖 `PreToolUse`、`PostToolUse`、`SessionStart/End`、`Stop` 等关键生命周期。`feature-dev` 插件的 7 阶段工作流（Discovery → Exploration → Questions → Design → Implementation → Review → Summary）是复杂任务分解的工程范本。参见 [[Understand-Anything]]、[[code-review-graph|code-review-graph]] 与 [[Kilo-Code|Kilo Code 插件]]。

## Kilo Code 生态深度解析

### 项目定位与架构全景

[[Kilo-Code|Kilo Code]] 是一个开源的"全栈智能体工程平台"（All-in-One Agentic Engineering Platform），采用 VS Code 扩展 + CLI + Cloud 的三位一体架构。其上游血缘清晰：VS Code 扩展 fork 自 Roo Code（Roo Code fork 自 Cline），CLI fork 自 OpenCode。

核心能力包括：500+ AI 模型集成（通过 [[Vercel-AI-SDK]] 抽象层）、5 种内置模式（Ask / Architect / Code / Debug / Orchestrator）、Memory Bank 长期记忆机制、MCP 服务器市场、Agent Manager 多智能体编排（基于 Git Worktree 隔离）。详见 [[Kilo-Code-代码生成率归因]]、[[KiloClaw-安全白皮书]] 与 [[KiloCode-AGENTS.md]]。

### 智能体运行时深度剖析

[[KiloCode-AGENTS.md|Kilo Code AGENTS.md]] 揭示了 CLI 层的核心模式：

- **命名空间模块**：TypeScript Namespace 组织代码（非 Class），每个模块导出 Zod Schema + 类型 + 函数
- **`Instance.state()`**：基于 `AsyncLocalStorage` 的项目级懒加载单例
- **`fn(schema, callback)`**：Zod 输入验证包装器
- **`Tool.define(id, init)`**：统一工具定义模式
- **`BusEvent.define()` + `Bus.publish()`**：进程内发布/订阅事件系统
- **Hono HTTP Server**：提供 REST API + SSE 实时通信
- **文件系统 JSON 存储**：`Storage.write(["session", id], data)` 路径数组键

运行时采用客户端-服务器模型：Hono HTTP 服务器处理请求，CLI TUI、VS Code 扩展、桌面应用（Tauri）、Web UI（SolidJS）作为客户端通过 HTTP + SSE 通信。

### 安全架构：KiloClaw 白皮书

[[KiloClaw-安全白皮书|KiloClaw 安全白皮书]] 展示了托管式 AI 智能体平台的纵深防御设计：

1. **五层租户隔离**：基于身份的路由（JWT + per-user pepper）→ 每用户 Fly 应用程序 → WireGuard 网络隔离 → Firecracker 虚拟机（与 AWS Lambda 同源的硬件虚拟化）→ 专用 LUKS 加密卷
2. **故障闭合身份验证**：数据库不可访问时拒绝所有请求，而非回退到不安全模式
3. **全面输入验证**：SQL 注入、XSS、命令注入（72 个对抗性负载测试）、路径遍历、开放重定向全覆盖
4. **数据生命周期管理**：实例销毁触发两阶段清理（Fly 机器销毁 + 卷删除 + LUKS 密钥销毁）

独立安全评估使用 PASTA 框架，覆盖 13 项资产中的 30 个威胁，执行 60+ 对抗性测试，验证了架构的健全性。参见 [[Kilo-Code|Kilo Code 架构]]、[[AI-智能体架构|智能体安全]] 与 [[Claude-Code|Claude Code 安全]]。

### 代码归因系统设计

[[Kilo-Code-代码生成率归因|Kilo Code AI 代码生成率与归因分析]] 提出了一套完整的 AI 代码贡献量化方案，解决"行数计数法"无法追踪修改、无法追踪存活、无法识别重排的三大缺陷。

核心设计：

- **AST-aware MinHash 指纹**：对 AST 节点类型序列做 k-Shingle（k=3），变量重命名不影响签名
- **LSH 分桶索引**：128 维签名分 16 个 band，每个 band 8 行，百万级指纹库毫秒级召回
- **三路径统一采集**：Tab（字符级 shingle）、Chat（文本级 shingle）、Agent（子任务回传）
- **行级 Diff 归因**：`AI 贡献占比 = (保留行 × 1.0 + 修改行 × 0.5) / 总行数`
- **置信度分级**：HIGH (>0.85) / LOW (0.55~0.85) / SUSPECT (0.30~0.55) / UNIQUE (<0.30)

实施路线图分 4 个 Phase：指纹采集（2 周）→ 指纹库存储（1 周）→ 归因引擎（2 周）→ 看板与 API（2 周）。参见 [[代码归因系统设计]]、[[MinHash-文本去重|MinHash 算法]] 与 [[Kilo-Code|Kilo Code 架构]]。

## 代码图谱与知识工程

### Understand-Anything：多智能体代码图谱

[[Understand-Anything|Understand-Anything]] 是一个 Claude Code 插件，通过 6 个智能体（project-scanner → file-analyzer → architecture-analyzer → tour-builder → graph-reviewer → domain-analyzer）的并行流水线，将代码库转化为交互式知识图谱。

核心输出是 `knowledge-graph.json`，包含：
- **节点**：文件、配置、文档三类，附 summary / tags / complexity
- **边**：imports / depends_on / deploys / configures / documents / tested_by 六种关系
- **架构层**：自动识别的层级结构（如 Game Core → Game Utilities → Electron Desktop → Test Layer）
- **导览步骤**：为新团队成员生成的学习路径

支持增量更新——仅重新分析变更文件。`/understand-dashboard` 命令生成交互式可视化看板，`/understand-chat` 支持自然语言问答。相关技术参见 [[代码归因系统设计|AST MinHash 指纹]]、[[MinHash-文本去重|MinHash 算法]] 与 [[Claude-Code|Claude Code 插件]]。

### code-review-graph：增量知识图谱

[[code-review-graph|code-review-graph]] 是一款本地代码知识图谱工具，基于 Tree-sitter 解析代码结构，通过 MCP 协议为 AI 编程助手提供精准最小上下文。

核心优势：
- **Token 用量降低 5–10 倍**：用增量图谱替代全量读码
- **零配置开箱即用**：`pip install code-review-graph` → `code-review-graph install --platform claude-code`
- **持久化增量图谱**：Schema 版本 1→9 的迁移历史（signature 列 → flows 表 → FTS5 全文索引 → 社区检测 → 风险指数）
- **图谱可视化**：`code-review-graph visualize` 生成交互式 HTML
- **Wiki 生成**：`code-review-graph wiki` 按社群结构生成 Markdown 知识库

关键 MCP 工具包括：`detect_changes`（风险评分变更分析）、`get_review_context`（Token 高效审查）、`get_impact_radius`（变更影响半径）、`query_graph`（调用者/被调用者/导入/测试关系追踪）。参见 [[MCP-协议生态]] 与 [[Understand-Anything|Understand-Anything]]。

## 文本相似度与代码归因

### 从 Jaccard 到 MinHash 的完整指南

[[MinHash-文本去重|海量文本去重与相似度检索]] 深入拆解了 MinHash 算法的数学原理与工程实现：

- **Jaccard 相似度**：`J(A,B) = |A∩B| / |A∪B|`，直觉陷阱是容易混淆"局部重合"与"全局重合"
- **MinHash 核心定理**：`P(min(h(A)) = min(h(B))) = J(A,B)`——两个集合最小哈希值相同的概率恰好等于 Jaccard 相似度
- **签名长度选择**：N=128 时，95% 置信区间宽度 ±0.087，是工业界的黄金平衡点
- **LSH 分桶策略**：128 维签名分 16 个 band × 8 行，将 O(N²) 全量比对降为候选对召回
- **工程实现**：MurmurHash3 + Python `mmh3` 库，预处理 O(N·M) + 比对 O(N)
- **应用场景**：参见 [[代码归因系统设计]] 与 [[Kilo-Code-代码生成率归因]]

相关应用：[[代码归因系统设计|代码归因]]、[[Kilo-Code-代码生成率归因|Kilo Code 归因系统]]、[[code-review-graph|code-review-graph]]。

### 代码归因与贡献占比设计

[[代码归因系统设计|AI 编码助手代码归因与贡献占比量化系统设计]] 将 MinHash 应用于代码场景，提出 AST-aware MinHash 方案：

- **AST Shingling**：前序遍历收集节点类型序列，生成 k=3 的 n-gram 集合
- **变量重命名免疫**：`x` → `inputValue` 后 AST 节点类型序列不变，签名不变
- **行级归因算法**：保留行算 100% 贡献，修改行算 50%，新增行算 0%
- **业务阈值**：HIGH (90~100%) / LOW (行级加权) / SUSPECT (仅标记) / UNIQUE (不计入)

关键落地建议：使用 tree-sitter 支持 40+ 语言、指纹库设置 90 天 TTL、按项目/团队分片存储、隐私合规上只存 AST 节点类型序列而非原始代码。参见 [[代码相似度算法]]、[[AI-智能体架构]] 与 [[MinHash-文本去重]]。

## 浏览器原生能力实战

### Web Speech API：TTS 开发经验

[[浏览器-TTS-开发经验|浏览器原生语音合成（TTS）开发经验]] 总结了两个纯前端朗读工具（Read Aloud 和智能朗读阅读器）的踩坑记录：

- **语音列表异步加载**：`getVoices()` 首次调用常返回空数组，需监听 `voiceschanged` 事件 + 超时兜底
- **多层回退语音选择**：用户选择 → 语言过滤 → 本地语音优先 → 浏览器偏好 → 默认语音
- **中英文自动检测**：按汉字/字母比例判断语言，避免混读
- **长文本分句**：单条 utterance 有长度限制，需按标点切分（300~800 字符上限）
- **状态管理**：区分"未播放 → 开始""播放中 → 暂停""已暂停 → 继续"三种操作
- **浏览器差异**：Safari 的 pause/resume 不可靠，需用 cancel + 重新 speak 替代，参见 [[浏览器-API-开发]] 与 [[QR-Code-笔记|前端编码经验]]
- **Watchdog 机制**：单句超过 18 秒无响应强制推进，处理浏览器不稳定行为
- **视觉反馈**：当前句高亮、已读淡化、进度条、滚动同步、点击跳转

参见 [[浏览器-API-开发]] 与 [[QR-Code-笔记|QR Code 前端编码]]。

### QR Code：前端编码踩坑

[[QR-Code-笔记|QR Code 调试笔记]] 记录了纯前端二维码工具的编码问题：

- **中文识别**：jsQR 的 `code.data` 对多字节支持不好，需用 `code.binaryData` + `TextDecoder('utf-8')` 解码
- **中文生成**：`qr.addData()` 前需先做 UTF-16 到 UTF-8 的字节转换
- **换行显示**：HTML 默认折叠空白，需 `white-space: pre-wrap`
- **响应式布局**：桌面端 flex 左右两栏，移动端 media query 恢复上下堆叠

参见 [[浏览器-TTS-开发经验|浏览器 API 开发]]、[[JSON-格式化工具|在线工具实践]] 与 [[Web-标准与协议]]。

[[llms.txt-文件|/llms.txt 文件]] 是 Jeremy Howard（fast.ai）提出的标准化提案，建议在网站根路径放置 `/llms.txt` Markdown 文件，为 LLM 提供精选的、结构化的网站内容概述。

规范要求：

## 知识表示新格式

### Open Knowledge Format (OKF)

[[Open-Knowledge-Format|Open Knowledge Format (OKF)]] 是一种开放、对人机友好的知识表示格式，由 Markdown 文件 + YAML 前置元数据组成。核心概念：

- **Knowledge Bundle**：自包含的层次化知识文档集合（分发单元）
- **概念文档**：YAML frontmatter（type/title/description/resource/tags/timestamp）+ Markdown 正文
- **交叉链接**：绝对路径（`/tables/orders.md`）与相对路径两种形式
- **索引文件**：`index.md` 支持渐进式展示
- **日志文件**：`log.md` 按日期分组记录变更历史
- **引用规范**：`# Citations` 章节编号列出外部来源

OKF 的合规性要求极简：每个非保留 `.md` 文件包含可解析的 YAML 块 + 非空 `type` 字段。消费者必须容忍未知类型、未知字段、断链——这种宽松模式支持 Bundle 增长、重构和部分由智能体生成。参见 [[知识表示格式]]、[[llms.txt-文件]] 与 [[DESIGN.md-规范]]。
