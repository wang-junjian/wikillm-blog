# 微调与模型训练

[[微调]]（Fine-tuning）是在预训练 [[大型语言模型]]（[[LLM]]）基础上，通过[[特定领域数据]]的[[继续训练]]，使模型适应[[专项任务]]的技术路径。与 [[Prompt-Engineering-与上下文工程]] 的"零参数更新"范式不同，微调直接修改[[模型权重]]，能够更深刻地改变模型的[[行为模式]]与[[知识边界]]。

## 技术定位

微调位于 [[LLM]] 应用开发的进阶层。当[[提示工程]]无法满足需求——需要改变模型的内在[[行为风格]]、注入大量[[领域知识]]、或实现特定任务的显著[[性能提升]]时，微调成为必然选择。其本质是一种[[迁移学习]]：保留[[预训练]]阶段习得的[[通用语言能力]]，在特定[[数据分布]]上进行[[参数适配]]。这与 [[RAG-检索增强生成]] 形成互补——RAG 注入外部知识，微调改变内在能力。

```mermaid
flowchart LR
    A[预训练模型] B[基座模型选择] C[数据集准备] D[训练方法选择] E[微调训练] F[评估验证] G[部署推理]
    A --> B --> C --> D --> E --> F --> G
    H[LoRA/QLoRA] I[P-Tuning v2] J[全量微调]
    H --> D
    I --> D
    J --> D
```

## 参数高效微调方法

### LoRA

[[LoRA]]（Low-Rank Adaptation）通过在冻结的[[原始权重]]旁注入[[低秩矩阵]]来实现[[参数高效微调]]。其核心假设是权重更新具有[[低秩特性]]——微调所需的参数变化可以用远少于原始[[参数量]]的[[矩阵]]来表达。

典型配置下，LoRA 仅训练模型总参数的 0.01%-5%。以 [[Mistral-7B]] 为例，LoRA 可训练参数仅 1.704M（约占总量的万分之 2.35），却能在 [[Text2SQL]] 等任务上取得良好效果。这种[[参数效率]]的提升使得在[[消费级硬件]]上微调[[大模型]]成为可能。

### QLoRA

[[QLoRA]]（Quantized LoRA）在 [[LoRA]] 基础上引入 [[4-bit 量化]]，大幅降低[[显存占用]]。这种[[量化]]策略与 [[bitsandbytes]] 库的结合，使得在单张 [[GPU]] 上微调 7B 模型成为现实。对比数据：

| 指标 | LoRA（Mistral-7B） | QLoRA（4-bit） |
|------|---------------------|-----------------|
| 总参数量 | 7243.436M | 1244.041M |
| 可训练参数 | 1.704M | 1.704M |
| 训练耗时（600 iter） | 20 分 26 秒 | 23 分 40 秒 |
| 融合模型大小 | 13G | 4G |
| 推理速度 | ~17 tok/s | ~107 tok/s |

核心权衡：QLoRA 的训练精度略低于 LoRA（Loss 1.396 vs 1.351），但融合后模型更小、推理速度提升约 6 倍，适合对内存和延迟敏感的场景。

### P-Tuning v2

[[P-Tuning v2]] 通过引入可训练的[[前缀编码器]]（prefix_encoder）来引导模型输出，属于[[前缀微调]]（Prefix-tuning）的改进版本。以 [[ChatGLM-6B]] 为例：

- 仅微调约 0.1% 的参数（prefix_encoder）
- pre_seq_len=128 的[[前缀提示]]
- 训练 3000 步即可在[[电力安全规程]]问答任务上获得良好效果
- 推理时 prefix_encoder 保持 float 精度，[[基座模型]]使用 half().cuda() 加速

P-Tuning v2 的核心价值在于[[参数量]]的极致压缩——相比[[全量微调]]，它仅更新极少量参数即可达到接近的[[任务适配]]效果。

## 微调框架

### LLaMA Factory

[[LLaMA Factory]] 是一个集成了多种微调方法的[[统一框架]]，通过 [[Web UI]]（LLaMA Board）极大降低了技术门槛。

**核心特性：**
- 支持多种主流 [[LLM]] 与训练方法（[[全量微调]]、[[P-Tuning v2]]、[[LoRA]] 等）
- 内置 [[Web UI]]，通过 `python src/train_web.py` 启动，浏览器访问 localhost:7860
- 从 [[ModelScope]] 魔搭社区下载[[模型]]和[[数据集]]
- 提供"[[微调模型]]"和"[[聊天]]"两大功能模块
- 支持 [[PEFT]] 库的多种[[适配器]]方法

**典型工作流：**
1. 配置模型路径与训练方法
2. 加载或上传数据集（支持 Alpaca 格式）
3. 设置训练参数（lora_rank、lora_alpha、learning_rate、epochs）
4. 启动训练并监控 Loss 曲线
5. 在聊天界面实时测试效果

### SWIFT

[[SWIFT]] 是 [[ModelScope]] 生态下的[[全流程]][[微调框架]]，支持近 200 种 [[LLM]] 和[[多模态大模型]]（MLLM）。

**核心特性：**
- 覆盖[[训练]]、[[推理]]、[[评测]]、[[部署]]全流程
- 内置 [[PEFT]] 轻量微调方案（[[LoRA]]、[[QLoRA]]、[[全量微调]]）
- 独立 [[Adapters]] 库，支持 [[NEFTune]]、[[LoRA+]]、[[LLaMA-PRO]] 等最新技术
- [[推理]]支持通过简单[[命令行]]指定[[模型类型]]和路径即可启动
- 支持 [[RLHF]] 与 [[DPO]] 等[[对齐]]方法

**与 LLaMA Factory 的对比：**

| 维度 | LLaMA Factory | SWIFT |
|------|--------------|-------|
| 定位 | 易用性优先，Web UI 驱动 | 覆盖面优先，200+ 模型 |
| 生态 | 独立框架，ModelScope 兼容 | ModelScope 原生 |
| 多模态 | 专注 LLM | LLM + MLLM |
| 扩展性 | 内置方法丰富 | 独立 Adapters 库 |

### MLX（Apple Silicon）

[[MLX]] 是 [[Apple]] 为 [[Apple Silicon]] 芯片设计的[[机器学习框架]]，利用[[统一内存架构]]（CPU/GPU 共享内存）运行[[大语言模型]]。

**核心优势：**
- [[MacBook Pro M2 Max]] 上即可完成 7B 参数模型的 [[LoRA]] 微调
- [[统一内存架构]]避免了 CPU-GPU 数据传输瓶颈
- 支持[[模型融合]]（fuse）与 [[GGUF]] 格式转换
- 无需 [[NVIDIA GPU]] 即可进行[[本地微调]]

**工具链：**
- `mlx-examples/lora`：旧版[[微调脚本]]
- `mlx_lm.lora` / `mlx_lm.fuse`：新版[[命令行工具]]
- `mlx_lm.generate`：[[推理生成]]

## 微调实践：命名实体识别（NER）

### 任务定义

[[命名实体识别]]（Named Entity Recognition，NER）是从[[非结构化文本]]中提取预定义类别[[实体]]（如人名、地名、机构名）的任务。在[[电力领域]]，NER 需要识别省/市/供电公司/供电所/指标/日期等实体。NER 是 [[NLP]] 的基础任务之一，也是评估 [[LLM]] 在[[领域任务]]上[[细粒度理解]]能力的试金石。

### 基于提示的 NER

使用 [[Qwen]] 等大模型，通过 [[LangChain]] 框架集成 [[DashScope]] API 进行[[零样本]] NER。实验对比了三种方案：

| 方案 | 格式 | 效果 |
|------|------|------|
| 实验一 | XML 标记 | 效果最佳 |
| 实验二 | 严格指令约束 | 略有提升 |
| 实验三 | JSON 格式（含位置信息） | 效果不如 XML |

**LLM 用于 NER 的固有局限：**
- [[生成能力]]导致[[内容膨胀]]（如擅自给地名加"市"后缀）
- [[指令遵循]]不足（新增未定义的实体类型）
- [[实体边界]]重叠
- [[位置偏移]]一旦发生，后续实体位置全部错误
- [[上下文长度]]受限，长文本处理能力不足

### 基于微调的 NER

通过微调让模型学会精确的[[实体标注]]，避免[[生成能力]]的干扰。

**数据集设计：**
- 基于真实的[[城市]]-[[供电公司]]-[[供电所]][[映射关系]]
- 按 6 种基础模板扩展为含/不含省、含/不含时间的 24 种变体
- 输出采用 [[XML]] 标记格式

**训练配置：**
- 模型：[[Qwen1.5]] 系列（0.5B、1.8B、4B）
- 框架：[[LLaMA Factory]] + [[LoRA]]
- 参数：lora_rank=8、lora_alpha=16、learning_rate=5e-05、epochs=2-3
- 采用[[监督微调]]（SFT）策略

**评估结果：**

| 模型 | 参数量 | 24 用例正确率 |
|------|--------|--------------|
| Qwen1.5-0.5B-Chat | 0.5B | 23/24 |
| Qwen1.5-1.8B-Chat | 1.8B | 24/24 |
| Qwen1.5-4B-Chat | 4B | 24/24 |

1.8B 模型即可在所有测试用例上表现完美，0.5B 仅在一个边界用例上出错（将"菏泽巨野县供电公司"整体识别为 Company 而非拆分为 City+Company）。

### 生成式 NER 微调

让模型不仅识别已有实体，还能在缺失时自动补全。例如用户输入"菏泽麒麟供电所"，模型自动补全省份和供电公司。

**数据集设计：**
- 12 种样本模板，覆盖省/市/供电公司/供电所/指标/日期的不同组合形式
- 使用 Qwen1.5-4B-Chat 进行 LoRA 微调

**挑战：**
- 对于"城市+供电所"或仅有"供电所"的输入，模型难以准确补全对应的供电公司和城市
- 建议条件允许时使用更大的 Qwen1.5-7B-Chat 进行微调

## 微调实践：Text2SQL

### 任务定义

[[Text2SQL]] 是根据给定的[[表结构]]（表名、列名）和[[自然语言问题]]生成对应 [[SQL]] 查询语句的任务。使用 [[WikiSQL]] 数据集进行训练与评估。这是评估 [[LLM]] 在[[结构化数据]][[理解]]与[[代码生成]]能力的经典任务。

### 数据预处理

[[WikiSQL]] 原始文本包含[[特殊 token]]，需要针对不同[[基座模型]]做差异化处理：

| 模型 | 预处理方式 |
|------|-----------|
| [[Mistral-7B]] | 去掉前 3 个字符 |
| [[Deepseek-Coder-7B]] | 去掉前 3 个和后 4 个字符，补上 `<｜end▁of▁sentence｜>` |
| [[TinyLlama]]-Chat | 套用[[对话模板]] `<|user|>...<|assistant|>` |

这种[[数据预处理]]的精细化程度直接影响[[微调]]效果——错误的 [[token]] 处理会导致模型无法正确[[终止生成]]。

数据格式示例：
```
table: <table_name>
columns: <col1>, <col2>, ...
Q: <question>
A: <SQL>
```

### 训练流程

完整的[[微调流水线]]包含以下步骤：

1. 运行 `data/wikisql.py` 生成 `train.jsonl`、`valid.jsonl`、`test.jsonl`
2. 使用 `python lora.py --model <model> --train --iters <N>` 训练
3. 使用 `--adapter-file adapters.npz --test` 评估测试集 [[Loss]] 和 [[PPL]]
4. 使用 `fuse.py` 将 [[LoRA]] adapter 融合回[[基础模型]]
5. 使用 `mlx_lm.generate` 生成 [[SQL]]

每一环节的[[数据格式]]一致性至关重要——训练、评估、生成必须使用相同的[[提示模板]]。

### 模型对比实验

| 模型 | 参数量 | 迭代次数 | 训练耗时 | 测试 Loss | 测试 PPL |
|------|--------|---------|---------|----------|---------|
| Mistral-7B (LoRA) | 7B | 600 | 20m26s | 1.351 | 3.863 |
| Mistral-7B (QLoRA) | 7B | 600 | 23m40s | 1.396 | 4.040 |
| Deepseek-Coder-7B | 7B | 600 | 19m35s | 1.473 | 4.364 |
| TinyLlama | 1.1B | 500 | 6m22s | - | - |

**关键发现：**
- 选择最优 [[checkpoint]] 而非最后一次迭代（如 [[LoRA]] 在 Iter 500 时 [[PPL]] 最低为 3.646）
- [[Deepseek-Coder]] 专为[[代码]]设计，[[SQL]] 生成能力强，中英文表现都很好
- [[Mistral-7B]] 对[[中文]]表名和问题的生成效果较差
- [[TinyLlama]] 训练极快但[[生成质量]]有限，适合[[原型验证]]

### 模型转换：MLX → GGUF

将 [[MLX]] 微调的模型转换为 [[GGUF]] 格式，可融入 [[llama.cpp]] 生态：

1. 使用 `fuse.py --de-quantize` 融合并反量化为 [[safetensors]]
2. 使用 [[llama.cpp]] 的 `convert.py` 转为 f32 GGUF
3. 使用 `quantize` 工具生成 Q8_0、Q4_1、Q4_0 [[量化]]版本

这种[[模型转换]]能力打通了[[训练]]与[[部署]]的壁垒——在 Mac 上微调，在任意设备上推理。

**量化精度对比：**

| 量化方式 | llama.cpp 速度 | MLX 速度 | 生成质量 |
|---------|---------------|---------|---------|
| Q8_0 | 39.37 tok/s | 17.58 tok/s | 最佳 |
| Q4_1 | 58.02 tok/s | 不稳定 | 可用 |
| Q4_0 | 最快 | 无意义文本 | 不可用 |

Q8_0 是最佳平衡点——llama.cpp 生成速度是 MLX 的 2 倍多，且生成质量稳定。

## 数据集制备工具

### Easy Dataset

## 微调方法论

### 数据质量优先

微调的效果上限由[[数据质量]]决定。[[ChatGLM-6B]] 微调实践中的关键经验：
- 数据应改写为[[人类对话]]形式
- 问题[[语义]]相近即可正确回答，无需精确匹配
- 推理时不能使用[[会话模式]]（避免[[上下文]]干扰）
- [[数据清洗]]与[[去重]]是不可忽视的前置步骤

### 模型选择策略

| 场景 | 推荐模型 | 理由 |
|------|---------|------|
| 高精度 SQL 生成 | Deepseek-Coder-7B | 代码专精，中英文能力强 |
| 推理速度优先 | QLoRA 量化 Mistral-7B | 4G 模型，107 tok/s |
| 资源受限（8GB 内存） | TinyLlama-1.1B | 1.1B 参数，6 分钟训练 |
| 中文 NER | Qwen1.5-1.8B-Chat | 1.8B 即可完美识别 |

### 训练参数调优

- **lora_rank**：通常 8-16，控制[[适配器]]容量
- **lora_alpha**：通常 16-32，控制[[学习率]]缩放
- **learning_rate**：5e-05 是常见起点
- **epochs**：2-5 轮，过多容易[[过拟合]]
- **batch size**：根据[[显存]]调整，[[梯度累积]]可模拟大 batch
- **max_length**：根据[[序列长度]]设置，影响[[显存占用]]

### 评估与迭代

- 不仅看 [[Loss]]，还要看 [[PPL]]（[[困惑度]]）和实际[[生成质量]]
- 选择测试集 [[PPL]] 最低的 [[checkpoint]] 进行融合
- 构建覆盖[[边界情况]]的[[评估集]]
- 人工抽检生成结果的[[正确性]]
- 使用 [[BLEU]]、[[ROUGE]] 等[[评估指标]]量化生成质量

## 模型优化与部署工具链

### ONNX Simplifier

[[ONNX Simplifier]]（onnxsim）是一款用于简化 [[ONNX]] 模型图的工具，通过消除冗余算子与[[常量折叠]]（constant folding）来优化推理效率。其核心能力由两部分组成：

- **常量折叠/传播**：消除结果恒为常量的算子
- **图变换**：通过调用 [[ONNX]] Optimizer 的各种 pass 实现算子融合（如 conv+bn 合并）

两种策略协同使用时能挖掘出大量隐藏的优化机会。例如 `add(add(x, 1), 2)` 经图变换为 `add(x, add(1, 2))` 后可通过常量折叠简化为 `add(x, 3)`。

**使用方式：**
```shell
onnxsim input_onnx_model output_onnx_model
```

值得注意的是，[[PyTorch]] 的 `torch.onnx.export` 默认开启 `do_constant_folding=True`，在导出 [[ONNX]] 时已完成大量优化。ONNX Simplifier 是对导出后模型的进一步增强，属于 [[微调]] 后[[部署]]阶段的关键优化工具。

## MLX 框架演进

### MLX 基础：统一内存架构

[[MLX]] 是 [[Apple]] 为 [[Apple Silicon]] 设计的[[数组框架]]（array framework），其核心创新在于[[统一内存模型]]——数组位于共享内存中，可在 CPU/GPU/Neural Engine 等任何支持设备上执行而无需数据传输。

**设计灵感来源：** [[NumPy]]、[[PyTorch]]、[[JAX]]、[[ArrayFire]]

**关键 API：**
- Python API 紧密对应 [[NumPy]]
- `mlx.nn`、`mlx.optimizers` 遵循 [[PyTorch]] 风格
- 提供完整的 C++、C、Swift API

### MLX 推理与量化实战

[[MLX]] 支持多种主流 [[LLM]] 的本地推理与量化：

| 模型 | 量化方式 | 模型大小 | Prompt 速度 | 生成速度 |
|------|---------|---------|------------|---------|
| Mistral-7B-Instruct-v0.2 | bfloat16 | 14GB | 43.1 tok/s | 19.4 tok/s |
| Mistral-7B-Instruct-v0.2 | float16 | 14GB | 37.4 tok/s | 20.5 tok/s |
| Mistral-7B-Instruct-v0.2 | int4 | 4GB | 30.1 tok/s | 52.6 tok/s |

**关键发现：** 4-bit 量化后模型体积缩减至 1/3（4GB），生成速度提升约 2.7 倍（52.6 vs 19.4 tok/s），但 Prompt 处理速度略有下降。这体现了[[量化]]在推理场景中的核心 Trade-off。

**支持的模型类型：**
- [[Phi-2]]：轻量级推理，适合[[原型验证]]
- [[Qwen]] 系列：需指定 `--trust-remote-code` 和 `--eos-token`
- [[Stable Diffusion]]：图像生成
- [[T5]]：序列到序列任务
- [[Whisper]]：语音识别

### MLX 新版工具链（mlx_lm）

[[MLX]] 的 `mlx_lm` 模块提供了更高层级的[[命令行工具]]，简化了[[微调]]与[[推理]]流程：

| 旧版工具 | 新版工具 | 功能 |
|---------|---------|------|
| `lora.py` | `mlx_lm.lora` | [[LoRA]]/[[QLoRA]] 微调 |
| `fuse.py` | `mlx_lm.fuse` | 模型融合 |
| `mlx_lm.generate` | `mlx_lm.generate` | 推理生成 |
| — | `mlx_lm.server` | HTTP 服务 |
| — | `mlx_lm.convert` | 模型转换与量化 |

新版 API 统一了参数风格，支持 `--adapter-file` 直接加载适配器进行推理，并内置了 [[HTTP]] 服务模式（`mlx_lm.server`），可通过 OpenAI 兼容的 `/v1/chat/completions` 接口访问模型。

## 微调知识体系

### Hugging Face NLP Course

[[Hugging Face]] NLP 课程系统讲解了 [[Transformers]] 的理论与微调实践，是理解 [[LLM]] 技术底座的核心学习资源。

**Transformer 模型分类：**

| 类型 | 代表模型 | 适用场景 |
|------|---------|---------|
| 编码器（Encoder） | [[BERT]]、RoBERTa、DistilBERT | 句子分类、[[命名实体识别]]、抽取式问答 |
| 解码器（Decoder） | [[GPT]]、GPT-2、CTRL | [[文本生成]] |
| 编码器-解码器 | [[T5]]、BART、Marian | 翻译、摘要、生成式问答 |

**核心概念体系：**
- **[[迁移学习]]**：在预训练模型基础上用特定任务数据[[微调]]，大幅降低数据与计算成本
- **[[预训练]]**：以[[自监督]]方式在大量原始文本上训练，无需人工标注
- **[[注意力层]]**（Attention layers）：Transformer 的关键特性，让模型关注句子中的相关词
- **架构 vs 检查点**：架构是模型的骨架（层与操作的定义），检查点是在架构中加载的权重

**[[微调]] vs 预训练的 Trade-off：**

| 维度 | 预训练 | 微调 |
|------|--------|------|
| 数据量 | 海量（TB 级） | 少量（千-万级样本） |
| 计算成本 | 数千 GPU 时 | 单 GPU 数小时 |
| 碳排放 | 高（相当于汽车全生命周期排放） | 极低 |
| 目标 | 通用语言能力 | 特定任务适配 |

课程强调：应始终尝试利用预训练模型并微调，而非从头训练。

### LLMOps

[[LLMOps]]（LLM Operations）是 [[MLOps]] 在 [[大模型]] 时代的延伸，专注于 [[LLM]] 全生命周期的运维与管理。[[DeepLearning.AI]] 推出的 LLMOps 短期课程涵盖了从模型选择、数据准备到部署监控的完整流程。

**LLMOps 的核心环节：**
- 模型选择与评估
- [[训练数据]]策划与标注
- 训练与调优
- 部署与监控
- 安全与合规

**自动化测试**是 LLMOps 的关键组成部分——通过自动化手段验证 [[LLM]] 输出的质量、安全性与一致性，确保模型在生产环境中的可靠性。

## 微调工程实践

### LLaMA-Factory 微调 Text2SQL（Qwen 版）

在 [[MLX]] 上使用 [[LoRA]] 微调 [[Mistral-7B]] 完成 [[Text2SQL]] 后，进一步在 GPU 环境下使用 [[LLaMA Factory]] 微调 [[Qwen1.5-4B-Chat]] 实现中文电力领域的 [[SQL]] 生成。

**训练配置：**
```yaml
model_name_or_path: qwen/Qwen1.5-4B-Chat
finetuning_type: lora
lora_target: q_proj,v_proj
dataset: text2sql
cutoff_len: 1024
max_samples: 3000
learning_rate: 0.0001
num_train_epochs: 3.0
per_device_train_batch_size: 2
gradient_accumulation_steps: 8
```

**关键经验：** `max_samples` 设置为 3000 效果不错，默认 500 太少会导致模型无法有效识别完整的"供电所"实体。这体现了[[数据量]]对[[微调]]效果的直接影响。

**数据集设计：** 采用 [[Alpaca]] 格式，instruction 包含完整的表结构（CREATE TABLE SQL），output 为对应的查询语句。

**完整工作流：** 训练 → 推理测试 → 合并（export）→ 推理合并后模型。[[LLaMA Factory]] 提供了 `llamafactory-cli` 统一接口管理整个流程。

### 华为 Atlas A2 NPU 微调

在华为 [[Atlas A2]]（Ascend 910 NPU）上使用 [[LLaMA Factory]] 进行 [[LoRA]] 微调，验证了框架对国产[[异构计算]]平台的支持能力。

**环境配置：**
- 计算平台：8 × Ascend 910（ARM 192 核，768GB 内存）
- 模型：[[Qwen1.5-7B-Chat]]
- 并行策略：[[DeepSpeed]] Zero-3

**关键适配：**
- NPU 不支持 `bfloat16`，需将模型配置修改为 `float16`
- 环境变量 `ASCEND_HOME_PATH` 指向 CANN 工具包
- 使用 `ASCEND_RT_VISIBLE_DEVICES` 指定 NPU 设备

**训练参数：** `lora_target: all`、`learning_rate: 1e-4`、`num_train_epochs: 1`、`per_device_train_batch_size: 1`、`gradient_accumulation_steps: 2`。

这一实践验证了 [[LLaMA Factory]] 在 [[NVIDIA GPU]] 之外的[[国产 AI 芯片]]上的可用性，对[[信创]]领域的 [[LLM]] 落地具有参考价值。

### Gradio DataFrame 辅助工具

[[Gradio]] 的 `DataFrame` 组件可用于构建 [[Text2SQL]] 数据的交互式可视化界面。通过 `gr.Dataframe` 加载 CSV 数据（包含 Text-SQL 对），配合 `gr.SelectData` 事件实现行选择联动——点击表格行后在 Textbox 中显示对应的自然语言问题与 [[SQL]] 语句。

这种轻量级 UI 工具在[[微调]]过程中可用于数据浏览、结果验证和演示，是 [[LLaMA Board]] 之外的轻量替代方案。

## 数据集与合成数据

### LLM 训练数据集生态

[[LLM]] 的训练依赖于大规模、多样化的[[数据集]]。主流预训练数据集的规模对比：

| 数据集 | 规模（Tokens） | 特点 |
|--------|--------------|------|
| FineWeb | 15T | 高质量网页过滤 |
| RedPajama2 | 20T | 多源混合 |
| Dolma | 3T | 含元数据 |
| SlimPajama | 627B | 去重清洗 |
| RefinedWeb | 500B | Falcon 训练数据 |
| The Pile | 340B | 多领域混合 |
| C4 | 172B | Common Crawl 子集 |

**代码数据集的特殊处理：** 结合 [[TF-IDF]] 或 [[BM25]] 改进代码检索效果，使用 [[TreeSitter]] 或 [[AST]] 进行语法分析构建更好的交互体验，采用 [[Jaccard 相似度]]检测代码重复。

### 合成数据

[[合成数据]]（Synthetic Data）是通过 [[LLM]] 生成的训练数据，已成为[[预训练]]和[[微调]]数据的重要来源。[[微软]] [[Phi]] 系列模型的成功推动了合成数据领域的发展。

#### Cosmopedia

[[Cosmopedia]] 是目前最大的开放合成数据集，由 [[Mixtral-8x7B-Instruct-v0.1]] 生成，包含超 3000 万个文件、250 亿个词元。其目标是重现 [[Phi-1.5]] 的训练数据。

**技术栈：**
- **主题聚类**：使用 `text-clustering` 对数百万 Web 样本聚类为 145 个簇
- **大规模生成**：使用 `llm-swarm` 部署 Mixtral，总计算超 1 万 GPU 时
- **基准去污**：10 词元重叠率 + `difflib.SequenceMatcher` 排除污染样本
- **训练工具**：`datatrove`（去重分词）、`nanotron`（训练）、`lighteval`（评估）

**提示策划要点：**
- 80% 以上提示来自网络数据（RefinedWeb 等）
- 为不同受众（少儿、专业人士、高中生）设计差异化提示
- 引入 UltraChat 和 OpenHermes2.5 补充常识知识
- 迭代优化避免模型生成重复的引导语

#### Distilabel + Argilla

[[Distilabel]] 是合成数据与 AI 反馈框架，[[Argilla]] 是协作式数据集标注工具。两者配合构成[[微调]]数据生产的完整流水线：

1. Distilabel 负责基于 [[LLM]] 的大规模数据生成
2. Argilla 负责人工审核与质量把控
3. 支持本地 [[LLM]]（[[Ollama]]、[[LM Studio]]、Jan）和云端推理（[[TGI]]、[[vLLM]]）

合成数据代表了[[数据集制备]]范式的转变——从人工标注转向 [[LLM]] 辅助生成，大幅降低了[[微调]]的[[数据门槛]]。

## 分布式训练与推理优化

### 多 GPU 训练技术栈

[[李宏毅]] 2025 课程系统介绍了多 GPU 训练 [[LLM]] 的核心技术：

- **[[DeepSpeed]]**：微软的[[分布式训练]]优化库，支持 ZeRO 系列并行策略（ZeRO-1/2/3）、CPU/NVMe offload、混合精度训练
- **[[Flash Attention]]**：通过 IO 感知的精确注意力算法，减少 GPU HBM 与 SRAM 之间的数据传输，显著加速训练
- **[[Liger Kernel]]**：针对 [[LLM]] 训练的 GPU 内核优化，融合多个算子减少内核启动开销
- **[[量化]]**：训练后量化（PTQ）与量化感知训练（QAT），降低模型部署成本

这些技术的组合使得在有限[[硬件]]资源下训练[[大模型]]成为可能，是 [[LoRA]] 等[[参数高效微调]]方法之外的另一条降本路径。

## 推理模型训练前沿

### Sky-T1-32B-Preview

[[Sky-T1-32B-Preview]] 是由 UC Berkeley NovaSky 团队开发的开源[[推理模型]]，在数学和编码基准上表现与 o1-preview 相当，**训练成本不到 450 美元**（8 × H100，19 小时）。

**训练方法：**
1. **数据策划**：使用 [[QwQ-32B-Preview]] 生成推理数据，覆盖数学（AIME、MATH、NuminaMath）和编码（APPs、TACO）领域
2. **拒绝采样**：过滤 QwQ 回答错误的样本，数学用精确匹配，编码用单元测试验证
3. **格式重写**：使用 GPT-4o-mini 将 QwQ 输出重写为格式良好的版本（将 APPs 准确率从 25% 提升至 90%）
4. **微调**：对 [[Qwen2.5-32B-Instruct]] 进行 3 轮微调，学习率 1e-5，batch size 96，使用 [[LLaMA Factory]] + [[DeepSpeed]] Zero-3

**关键发现：**

| 发现 | 详情 |
|------|------|
| 模型规模 | 7B/14B 模型仅观察到适度改进，32B 是推理能力涌现的阈值 |
| 数据混合 | 数学与编码需平衡训练，单一领域数据会降低另一领域表现 |
| 训练效率 | 17K 数据 + 19 小时即可达到前沿推理水平 |

**评估结果：**

| 基准 | Sky-T1-32B | Qwen2.5-32B-Instruct | QwQ | o1-preview |
|------|-----------|---------------------|-----|-----------|
| Math500 | 82.4 | 76.2 | 85.4 | 81.4 |
| AIME2024 | 43.3 | 16.7 | 50.0 | 40.0 |
| LiveCodeBench-Easy | 86.3 | 84.6 | 90.7 | 92.9 |
| LiveCodeBench-Medium | 56.8 | 40.8 | 56.3 | 54.9 |

Sky-T1 证明了以极低成本复制高级推理能力的可行性，其完全开源（数据、代码、模型权重）的策略为学术界和开源社区提供了重要的[[后训练]]（post-training）参考。
