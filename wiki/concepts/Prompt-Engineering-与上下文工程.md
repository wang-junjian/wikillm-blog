# Prompt Engineering 与上下文工程

[[提示工程]]（Prompt Engineering）是驾驭 [[大型语言模型]]（[[LLM]]）的核心技艺——通过精心设计的指令、示例与上下文，引导模型在无需更新权重的前提下完成复杂任务。其本质并非"花哨的文字游戏"，而是一套系统性的方法论：将模糊的人类意图转化为模型可执行的精确规范，同时约束输出结构、注入领域知识、规避安全风险。

## 技术定位

提示工程处于 [[LLM]] 应用开发的第一层。在 [[微调与模型训练]] 的成本曲线之上，它代表了一种"零参数更新"的轻量范式：所有智能行为都通过输入侧的上下文来驱动。这与 [[RAG-检索增强生成]] 的外化思想互补——RAG 负责注入事实知识，提示工程负责编排推理过程。

```mermaid
flowchart LR
    A[人类意图] B[提示设计] C[上下文组装] D[LLM 推理] E[结构化输出] F[后处理/校验]
    A --> B --> C --> D --> E --> F
    G[示例库] H[领域知识] I[安全策略]
    G --> C
    H --> C
    I --> B
```

## 基础提示模式

### 零样本与少样本提示

[[零样本提示]]（Zero-shot Prompting）仅提供任务结构，依赖模型从预训练知识中推断意图。例如 [[情感分析|情感分类]]任务中，仅给出 `Message: ... Sentiment: ?` 的格式即可引导模型作答。其优势在于简洁，但面对细粒度任务时准确率有限。

[[少样本提示]]（Few-shot Prompting）在提示中嵌入若干已完成示例（通常为 2-5 个），让模型通过 [[上下文学习]]（In-Context Learning，ICL）捕捉任务模式。实践表明，示例的**质量与多样性**比数量更重要——覆盖边界情况的少量示例远胜于大量同质示例。[[Llama-2]] 在少样本场景下表现尤为突出，7B 与 70B 模型的能力差异可通过精心设计的示例缩小。

### 角色提示

[[角色提示]]（Role Prompting）为模型赋予特定身份（如"生活教练""资深软件工程师"），并指定回应语气与专业层级。[[Llama-2]] 在角色提示下通常给出更一致的响应。但需注意，角色提示也可能被恶意利用进行[[提示注入|越狱]]（Jailbreak），因此生产系统中应配合 [[#安全分类器 Llama Guard]] 使用。这种[[AI-安全|安全]]权衡是提示工程必须考量的维度。

### 输出格式控制

明确指定输出格式是提示工程的基本功。常用手段包括：

- **限定词约束**：如"给出一个词的回答""最多 50 个词"
- **结构化格式**：要求 JSON、HTML、Markdown 表格等
- **格式指令嵌入**：通过 [[LangChain]] 的 `StructuredOutputParser` 自动生成格式说明并注入提示

实践发现，中文提示中使用"词"作为计数单位比"单词"更自然；英文提示在生成 HTML 等结构化输出时质量往往更高。

## 思维链推理

[[思维链]]（Chain-of-Thought，CoT）是提升模型推理能力的关键技术。核心思想是：在要求模型给出最终答案之前，强制其输出中间推理步骤。

### 触发方式

- **指令触发**：在提示中加入"一步一步思考"（Let's think step by step）
- **示例触发**：在 few-shot 示例中展示完整的推理过程
- **格式约束**：要求"先解释推理，再给出答案"

### 指令顺序效应

LLM 逐 token 预测的特性意味着指令顺序会影响结果。实验表明，要求模型"先回答再解释"时，推理质量显著下降——因为模型一旦生成答案，后续解释往往只是合理化已有结论。最佳实践是**先推理后作答**，让中间步骤锚定最终输出。

### 复杂推理场景

对于[[数学推理|数学题]]、[[逻辑推理]]、[[多步决策]]等任务，思维链能将准确率提升 30%-50%。在 [[Building Systems with the ChatGPT API]] 课程中，通过让模型先自行解题再与学生答案对比，成功识别出学生将维护费错算为 100x（应为 10x）的错误。这种[[推理能力]]的提升在 [[Llama-2]] 70B 模型上尤为显著。

## 提示链与系统架构

### 链式处理

[[提示链]]（Chaining Prompts）将多个提示串联成流水线，前一步输出作为后一步输入。[[LangChain]] 框架提供了多种链式模式：

- **LLMChain**：基础的提示-模型绑定单元
- **SimpleSequentialChain**：单输入单输出的顺序链
- **SequentialChain**：支持多输入多输出的复杂顺序链
- **RouterChain**：根据输入内容动态路由到专业处理链

### 端到端系统架构

生产级 LLM 应用通常采用模块化架构：

```
用户输入 → 输入分类 → 输入审核 → 思维链推理 → 提示链组合 → 输出校验 → 最终回复
```

每个阶段由独立的提示和辅助函数完成，形成可组合、可测试的流水线。这种架构使得复杂任务可拆解、可维护，且每个环节可独立优化。

### 输入审核与安全

- **[[Moderation API]]**：调用内容审核接口检测仇恨、色情、暴力等 7 个维度
- **[[提示注入]]防护**：使用[[分隔符]]（如 `####`）隔离用户消息与系统指令
- **独立检测提示**：让 [[LLM]] 判断用户是否试图注入恶意指令
- **[[Few-shot]] 示例模式**：在提示中提供正例和反例，教会模型区分正常请求与注入攻击

## 上下文工程：超越单次提示

### 提供新信息

[[LLM]] 的知识截止于训练数据的时间点。对于训练后发生的事件（如 2023 年女足世界杯），模型无法通过参数记忆获知。提示工程通过**在提示中直接嵌入相关上下文**来弥补这一缺陷——这构成了 [[RAG-检索增强生成]] 的雏形。

### 上下文窗口管理

长对话场景中，上下文窗口成为稀缺资源。[[LangChain]] 的 Memory 模块提供多种策略：

| 记忆类型 | 策略 | 适用场景 |
|---------|------|---------|
| ConversationBufferMemory | 完整保存历史 | 短对话 |
| ConversationBufferWindowMemory(k) | 保留最近 k 轮 | 中等长度对话 |
| ConversationTokenBufferMemory | 按 token 数限制 | 精确控制成本 |
| ConversationSummaryMemory | LLM 生成摘要 | 长对话 |
| ConversationSummaryBufferMemory | 混合策略 | 兼顾连续性与效率 |

### 迭代式提示开发

提示工程不是一次性写作，而是持续迭代的过程。典型迭代路径：

1. **初版提示** → 观察输出
2. **添加约束**（字数、格式、聚焦点）→ 再次测试
3. **边界情况处理** → 覆盖异常输入
4. **系统化评估** → 构建测试集验证鲁棒性

以"从产品说明书生成营销描述"为例，经过三轮迭代：限制字数 → 明确受众与聚焦点 → 添加结构化格式要求，最终获得可用输出。

## 模型能力与局限

### 核心能力

- **总结**：控制字数、聚焦角度、批量处理
- **推理**：情感分析、主题提取、多任务并行
- **转换**：语言翻译、语气转换、格式转换
- **展开**：基于种子信息生成扩展内容

### 幻觉问题

[[幻觉]]（Hallucination）是 [[LLM]] 的核心局限——模型会生成流畅可信但完全编造的内容。以"Boie 公司的 AeroGlide UltraSlim 智能牙刷"为例：Boie 是真实公司，但该产品并不存在，模型却生成了详尽的产品介绍。缓解策略包括：

- 通过 [[RAG-检索增强生成]] 提供事实锚点
- 要求模型标注信息来源
- 使用 [[#输出校验]] 进行事实核查

### 温度参数

`temperature` 控制输出的[[随机性]]：

- **temperature=0**：确定性输出，适合[[分类]]、[[审核]]等任务
- **temperature=0.7**：适度多样性，适合[[创意写作]]
- **temperature=1**：最大创造性，但可能牺牲[[一致性]]

这一参数直接影响 [[LLM]] 输出的[[概率分布]]采样策略，是提示工程中调控创造性与稳定性的核心旋钮。

## 代码生成：Code Llama

[[Code Llama]] 是 [[Meta]] 推出的[[代码专用模型]]系列，支持[[代码生成]]、[[代码补全]]与[[调试]]。其特点包括：

- **[[代码生成]]**：根据[[自然语言描述]]生成可运行代码
- **[[代码填充]]**（In-filling）：补全部分完成的代码
- **[[长上下文窗口]]**：支持 20,000+ 字符，远超 [[Llama-2]] Chat 的 4097 token 限制
- **[[迭代优化]]**：通过"生成→批判→改进"三步法提升代码质量

实验显示，递归版[[斐波那契函数]]在 n=40 时耗时约 33.76 秒，而循环版仅需 0.00006 秒——模型能准确指出[[时间复杂度]]差异并主动优化。这体现了 [[LLM]] 在[[代码理解]]与[[算法优化]]方面的能力边界。

## 安全分类器：Llama Guard

[[Llama Guard]] 是专门训练的内容安全分类模型，用于检查对话中 User 输入或 Agent 输出是否包含不安全内容。

### 六类不安全内容

| 类别 | 说明 |
|-----|------|
| O1 | 暴力与仇恨 |
| O2 | 色情内容 |
| O3 | 犯罪计划 |
| O4 | 枪支与非法武器 |
| O5 | 管制物质 |
| O6 | 自残 |

### Prompt 结构

Llama Guard 的提示由四要素拼接而成：

1. **Task** — 声明检查角色（User/Agent）
2. **Policy** — 嵌入完整安全策略（不可修改，因模型专门训练过）
3. **Conversation** — 用标签包裹对话内容
4. **Output Format** — 要求首行输出 `safe`/`unsafe`，若 unsafe 则次行列出违规类别

### 越狱检测

通过"角色扮演 + 禁止提及角色名"的组合 trick，可让 Llama 2 7B Chat 输出敏感内容；Llama Guard 能正确将其标记为 `unsafe / O3`（犯罪计划）。这揭示了安全分类器在系统中的必要性。

## 评估方法论

### 模型评分评估

[[模型评分评估]]（Model-Graded Evaluation）让一个 LLM 评估其他 LLM 的输出。典型流程：

1. 提供原始文本、多个模型的名称及其输出
2. 要求评估者从准确性、指令遵循、其他特征三个维度打分
3. 综合比较并推荐最佳模型

### 系统化评估

- 构建覆盖边界情况的评估数据集
- 不仅评估最终答案，还评估中间步骤
- 自动化测试与人工抽检结合
- 注意提示的鲁棒性——对某个输入有效的提示不一定对所有输入都有效

## 实践要点

1. **[[分隔符]]隔离**：用 `####` 等分隔符区分[[系统指令]]与[[用户输入]]，防止[[提示注入|注入]]
2. **[[Few-shot]] 示例**：提供正例和反例，教会模型[[期望行为]]
3. **[[Token]] 计数**：通过 `tiktoken` 监控消耗，控制 `max_tokens` 参数
4. **[[模块化封装]]**：将提示、[[模型调用]]、[[输出解析]]封装为可复用函数
5. **[[中英文差异]]**：英文 Token 更少、成本更低；中文提示需更精细的措辞设计
6. **[[格式敏感性]]**：提示格式（如换行）的细微变化可能导致[[分类]]失效
7. **[[安全策略]]不可修改**：专用安全模型的配套提示是训练时固定的，擅自更改会破坏分类能力
8. **[[上下文学习]]质量**：示例的排列顺序与多样性直接影响 [[ICL]] 效果
9. **[[系统提示]]设计**：通过 system 角色设定模型人设，在多轮对话中维持[[一致性]]
